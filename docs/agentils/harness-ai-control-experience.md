# 在自己的仓库里搭一套让 AI 帮我写代码的开发流程

> 一个 maintainer 用 AgentILS 这个仓库做案例。
>
> **本文不讲 AgentILS 这个产品做什么**（`packages/` 下的代码功能不重要）。
> 只讲：**我在 AgentILS 这个仓库里部署了哪些工程基础设施，让 AI 能稳定帮我写出 `packages/` 下的代码而不至于失控**。
>
> 所有引用都指向仓库里实际存在的文件，可以打开核对。

---

## 一、为什么我会去搭这些东西 —— 真实踩过的坑

任何流程规范都不是凭空想出来的，是被打疼之后才长出来的。先把六件事的痛点起源写在前面，否则后面读起来会像方法论自嗨。

### 痛 1：来回切 agent，规则书来回撕裂

早期手上挂着 codex 和 copilot。codex 配额烧完切 copilot，copilot 上下文压满等 codex 配额恢复又切回去。

- codex 读的是 `AGENTS.md`。
- copilot 读的是 `.github/copilot-instructions.md` + `.github/instructions/*.md`。
- 我懒，文档只更其中一份。
- 结果：codex 还在按方案 A 改代码，copilot 已经按方案 B 改了。同一个文件来回拉锯，commit history 看着像精神分裂。
- 上下文一压缩，agent 开始**自己脑补已删除的旧 tool 名**（V0 时代散点架构留下的 `new_task_request` / `approval_request` / `feedback_gate`），错得理直气壮。

→ 长出 [缰绳 1：instructions sync 脚本](#1-instructions-sync-脚本--多入口同源永不漂移)。

### 痛 2：agent 把烂格式直接 push 上去

agent 写完代码顺手 `git commit && git push`：

- 没格式化。
- 没 lint。
- 类型 any 满天飞。
- commit message 一句 `update files`。

我 review PR 的时候要先帮它 prettier、再帮它收拾命名、再帮它写 changelog。**review 变成保姆**。

→ 长出 [缰绳 2：lint-staged + ECAM pre-commit panel](#2-lint-staged--husky-pre-commit--ecam-panel)。

### 痛 3：架构图与代码漂移

`docs/flowcharts/*.md` 是 mermaid 写的，agent 改了 task loop 的代码忘了改图。新人（包括下一次会话的"我"）读图照着错的画面改代码 → 越改越歪。

→ 长出 [缰绳 3：流程图自动出 PNG + pre-commit 同步检查](#3-流程图自动生成--文档与代码不漂)。

### 痛 4：commit log 不可读

agent 提的 commit message 经常是中英混杂、subject 一句动词、body 抄一遍 diff。一个月后回头看 git log，不知道哪个 commit 改了什么。

→ 长出 [缰绳 4：commitlint + 全英文 + 72 字符](#4-commitlint--conventional-commits--全英文--72-字符)。

### 痛 5：dev 分支让 PR 丢东西

最早试过 `feat/* → dev → main` 的两段流。结果：

- agent 开 PR target 选错，往 dev 推了，main 上没有。
- dev 合 main 时，dev 上有人 force push 过，**几个 commit 直接消失**，没人发现，下游 push 又把缺的填了一遍。
- 单人 / 双人开发下 dev 上根本没有真正的 staging 验证场景，只是制造同步负担。

→ 长出 [缰绳 5：GitHub Flow + branch protection + squash merge](#5-github-flow--branch-protection--squash-merge)。

### 痛 6：发布凭证泄露恐慌 + changelog 噪音

- 早期 npm publish 用 `NPM_TOKEN` secret，每次 agent 上下文里 echo 一下环境变量我都心跳加速。token 一旦被 agent 读到、被截图、被日志保存就完蛋。
- 早期 changelog 在 pre-push hook 里跑生成，每次 push 多一个 `chore: update changelog` 噪音 commit，分不清"已发布"和"待发布"。
- 升级 npm 11.5+ 想用 OIDC 时撞 bug：Node 22 + `npm install -g npm@latest` 会爆 `MODULE_NOT_FOUND 'promise-retry'`，bundled-deps 解析坏了，`--force` 救不了。

→ 长出 [缰绳 6：CI + changesets + npm OIDC](#6-ci--changesets--npm-oidc-trusted-publisher)。

## 二、六根缰绳（按部署顺序，每根对应上面一个痛）

### 1. instructions sync 脚本 —— 多入口同源、永不漂移

**真值源**：`docs/instructions/*.md`、`docs/skills/<dir>/SKILL.md`、`docs/instructions/sync-manifest.json`。

**生成器**：[`scripts/dev/sync-agent-instructions.mjs`](../../scripts/dev/sync-agent-instructions.mjs)。把真值源 fan-out 到所有 agent 实际读取的入口：

```
docs/instructions/foo.instructions.md   ──┐
docs/skills/bar/SKILL.md                  ├── 真值源 (人改这里)
docs/instructions/sync-manifest.json    ──┘
                |
                v   pnpm sync:instructions
                |
.github/instructions/foo.instructions.md     ← Copilot 读
.github/skills/bar/SKILL.md                  ← Copilot 读
.agents/skills/bar/SKILL.md                  ← Codex 读
.github/copilot-instructions.md              ← Copilot 入口
AGENTS.md                                    ← Codex 入口
```

每份生成文件顶部带 banner：

```
<!-- Generated by scripts/dev/sync-agent-instructions.mjs. -->
<!-- Source: <docs/...> -->
<!-- DO NOT EDIT this generated target. Edit the source file instead. -->
```

**CI 检查**：[`.github/workflows/ci.yml`](../../.github/workflows/ci.yml) 步骤 `Verify instruction sync (no drift)` 跑 `pnpm sync:instructions:check`，agent 手改任何镜像 → 红灯。

**配套 skill**：[`docs/skills/instructions-sync-discipline/SKILL.md`](../skills/instructions-sync-discipline/SKILL.md)，列出禁止行为（手改 `.github/instructions/*`、跳过 `sync-manifest.json` 注册新文件、用 `--no-verify` 绕过 pre-commit `--check`）。

**收益**：换 agent / 换 IDE / 新会话 / 上下文压缩 —— **agent 看到的规则永远是同一份**。痛 1 消灭。

### 2. lint-staged + husky pre-commit + ECAM panel

**hook 文件**：

```sh
# .husky/pre-commit
node packages/quality-gate/dist/precommit.js

# .husky/commit-msg
pnpm exec commitlint --edit "$1"
```

**lint-staged 配置**（[`lint-staged.config.mjs`](../../lint-staged.config.mjs)）：

```js
{
  '*.{ts,tsx,js,mjs,cjs}': ['eslint --fix --max-warnings=0 --no-warn-ignored', 'prettier --write'],
  '*.{json,md,yaml,yml}': ['prettier --write'],
}
```

`--max-warnings=0` 是关键 —— 一条 warning 都过不去，agent 不能用 `// eslint-disable-next-line` 凑合。

**ECAM panel**：`packages/quality-gate/dist/precommit.js` 的渲染层借自 A320 飞机的 ECAM（Electronic Centralized Aircraft Monitor）面板形态，把 "instructions sync → 流程图生成 → lint-staged" 三步用旋转指示 + 颜色块呈现，红了一目了然。详见 [`docs/instructions/quality-gate.instructions.md`](../instructions/quality-gate.instructions.md)。

**绕不过去**：

- `--no-verify` 会在 `git reflog` 留痕，PR review 一眼看得到。
- 即使 agent 绕过本地，CI 同样跑 lint / typecheck / sync:check，同一份规则在 GitHub 再拦一次。

**收益**：commit 阶段就拦，**烂代码进不了仓库**。痛 2 消灭。

### 3. 流程图自动生成 —— 文档与代码不漂

**源**：`docs/flowcharts/01-plan-c-topology.md`、`02-v1-task-loop.md` 等 mermaid 文件。

**脚本**：[`scripts/flowcharts/generate-all.mjs`](../../scripts/flowcharts/generate-all.mjs)，调 mmdc 出 PNG。

**约束**：pre-commit 检测到 `.md` 改了但对应 PNG 没动 → 红。

**为什么不用浏览器渲染 mermaid 算了**：

- PR review 在 GitHub web 上要等 mermaid 加载，慢且不稳定。
- 移动端 / 邮件通知里 mermaid 不渲染，PNG 永远渲染。
- 强制"图源改了 → PNG 必更"等于强制 agent 真的去看流程图，而不是只改代码。

**收益**：架构图永远是**当前**代码的图，agent 改流程**忘了画图就 commit 不下去**。痛 3 消灭。

### 4. commitlint + conventional commits + 全英文 + 72 字符

**配置**（[`commitlint.config.mjs`](../../commitlint.config.mjs)）：

```js
{
  extends: ['@commitlint/config-conventional'],
  rules: {
    'type-enum': [2, 'always',
      ['feat','fix','docs','style','refactor','perf','test','build','ci','chore','revert']],
    'subject-case': [2, 'never', ['sentence-case','start-case','pascal-case','upper-case']],
    'subject-max-length': [2, 'always', 72],
    'subject-empty': [2, 'never'],
    'type-empty': [2, 'never'],
    'scope-case': [2, 'always', 'lower-case'],
  },
}
```

**配套 branch 命名**（[`docs/skills/branch-name-standard/SKILL.md`](../skills/branch-name-standard/SKILL.md)）：分支前缀必须 = commit type，例如 `docs/harness-ai-control-experience` ↔ `docs(agentils): ...`。**branch 名 + commit type + PR 标题三者保持一致**。

**全英文规则**写在 [`docs/instructions/agentils.instructions.md`](../instructions/agentils.instructions.md) `## 文档语言规范`：subject + body 全英文。中文在 commit message 里 GitHub UI / 邮件通知 / git log -- 显示都会乱码或难搜。

**收益**：git log 是可读历史，可以按 type 过滤、按 scope 检索、可被外部 reviewer 跨地域复用。痛 4 消灭。

### 5. GitHub Flow + branch protection + squash merge

**唯一长期分支**：`main`。

**任何改动**：`<type>/<short-kebab>` → PR → squash merge。GitHub branch protection：

- 禁直推 `main`。
- PR 需要 CI 全绿。
- PR 需要 reviewer。
- 合并方式只允许 squash（disable merge commit + disable rebase merge）。

**为什么 squash**：

- `main` 历史 = "一个 PR 一个语义化 commit"，回滚 = `git revert <sha>`，简单可靠。
- agent 在分支里 `wip / fix typo / oops` 怎么乱提都行 —— squash 后只剩 PR 标题那一行进 main。

**为什么不再有 dev**：见痛 5。规则写在 [`docs/instructions/agentils.instructions.md`](../instructions/agentils.instructions.md) `## 分支与 PR 流` 里，包含历史背景"早期试过 feat → dev → main 的线性流，结果 ..."，让所有 agent 都看得到，不会再有人提"我们要不要加个 dev 分支"。

**配套 CI 改动检查**：[`.github/workflows/ci.yml`](../../.github/workflows/ci.yml) 里 `Verify changeset present` 步骤 —— 改了可发布包但没带 `.changeset/*.md` → 红：

```yaml
CHANGED_PKG=$(git diff --name-only "$BASE_SHA" "$HEAD_SHA" -- 'packages/**' \
  | grep -v '^packages/extensions/' \
  | grep -v '^packages/mcp/' \
  | grep -v '^packages/cli/' \
  || true)
if [ -z "$ADDED_CHANGESET" ]; then
  echo "::error::This PR modifies publishable packages/** but adds no .changeset/*.md."
  exit 1
fi
```

agent 改 `@agent-ils/quality-gate` 想偷偷不写 changeset → CI 直接拒。

**收益**：所有路径单一可追，PR 不丢东西，回滚廉价。痛 5 消灭。

### 6. CI + changesets + npm OIDC Trusted Publisher

**[`ci.yml`](../../.github/workflows/ci.yml)** —— Node 22 LTS：
`build → typecheck → lint → sync:instructions:check → changeset 存在性`。
顺序固定：typecheck 必须在 build 后，因为 workspace 间类型解析依赖 d.ts；turbo `typecheck.dependsOn` 包含 `^build`。

**[`release.yml`](../../.github/workflows/release.yml)** —— Node **24**（不是 22）：

```yaml
permissions:
    contents: write
    pull-requests: write
    id-token: write # ← OIDC 关键
- uses: actions/setup-node@v6
  with:
      node-version: 24 # ← Node 24 自带 npm 11.5+，OIDC 最低要求
      registry-url: https://registry.npmjs.org
- uses: changesets/action@v1
  with:
      version: pnpm changeset version
      publish: pnpm changeset publish
  env:
      GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
      # No NPM_TOKEN: this repo uses npm Trusted Publisher (OIDC).
      NPM_CONFIG_PROVENANCE: 'true'
```

**为什么 release 单独用 Node 24**（这条规则是踩坑后写进 [`docs/instructions/agentils.instructions.md`](../instructions/agentils.instructions.md) `## CI` 的）：Node 22 + `npm install -g npm@latest` 会爆 `MODULE_NOT_FOUND 'promise-retry'`，bundled-deps 解析有 bug，`--force` 也救不了。Node 24 直接自带 npm 11.5+，跳过 self-replace。

**OIDC 替代 token**：仓库 secrets 里**没有 NPM_TOKEN**。每个可发布包（仅 `@agent-ils/quality-gate`、`@agent-ils/logger`，其它包标 `"private": true`）在 npm 注册 Trusted Publisher，指向本仓库 + `release.yml` workflow，OIDC 流程自动换短期 token + `--provenance` 签名。**agent 看不到也偷不到**。

**Version PR 模式**：

- agent 写 `pnpm changeset` 生成 `.changeset/<name>.md`，跟代码一起提交。
- PR 合并到 main → `changesets/action` 自动开 `chore(release): version packages` PR。
- Version PR 合并 → 自动 `pnpm changeset publish` + 打 git tag + 推送。
- maintainer 只 review + merge Version PR，**不在本地跑任何发布命令**。

**为什么 changelog 不在 pre-push 跑**（这条也是踩坑后写进 instructions 的）：会污染 commit 范畴、产生噪音 chore 提交、模糊"已发布"语义。已废弃的脚本（`changelog` / `generate:changelog*`）和 `.husky/pre-push` 不要复活。

**收益**：发布零人工、零长期凭证、零噪音 commit。痛 6 消灭。

## 三、为什么这六件事的顺序不能换

| 顺序 | 缰绳         | 拦截点              | 为什么必须在它之前的都先布好            |
| ---- | ------------ | ------------------- | --------------------------------------- |
| 1    | instructions | agent 读规则的瞬间  | 没读到规则，下面五条 agent 都不知道存在 |
| 2    | pre-commit   | 本地 commit 之前    | 本地拦最便宜，CI 不浪费                 |
| 3    | 流程图       | 跟 commit 一起      | 文档与代码同步是 commit 语义的一部分    |
| 4    | commit msg   | commit 完成时       | 历史层防线                              |
| 5    | branch + PR  | push 后、merge 前   | 合并层防线                              |
| 6    | CI + 发布    | 合到 main 后、对外前 | 最后一道，"从仓库出去"那一步           |

漏一层后面就要替它擦屁股。常见错误：跳过 1，靠 5 和 6 的 CI 拦 agent。结果 agent 不知道有规则、写了一坨 → CI 红 → 它继续猜 → 反复红，烧 token、烧时间。**最贵的拦截点是 CI，最便宜的拦截点是 instruction**。

## 四、五条工程哲学（沉淀在所有六根缰绳里）

1. **约束写进 hook / CI / 配置文件，不写进对话**
   写在 prompt 里的约束，agent 上下文一压就丢；写在 hook 里的约束，agent 永远绕不过。
2. **每种失控模式对应一根缰绳；没缰绳别给新自由**
   想给 agent 一项新权限（让它能自动发版、能自动改 dependency 版本）→ 先想清楚怎么拦它滥用，再开权限。
3. **可观测 > 可解释**
   不要追求"agent 解释它为什么这么改"（它会编故事）。要追求"留下 git reflog / CI log / CHANGELOG / Version PR"，让事后能还原。
4. **可中止 > 可正确**
   宁可让 agent 在 commit / push / publish 任意一步被拦下来 retry，也不要一次性放过去出问题再追溯。
5. **凭证永远不进 agent 上下文**
   能用 OIDC 的别用 token，能用 short-lived 的别用 long-lived，agent 看不到的最安全。

## 五、什么仓库适合 / 什么仓库不适合

**适合**：

- 长期维护、有真实 release 节奏的项目。
- 团队 + AI 协作，或单人但希望 AI 反复接管。
- 关心代码、文档、发布物全链路质量的人。

**不适合**：

- 一次性脚本、PoC、玩具仓库（搭这套基础设施的成本回收不回来）。
- 只想让 AI 帮忙生成片段然后手动粘贴的人（用聊天框就够了）。
- 完全不想读 instruction、希望 AI 一次猜对的人（这套体系会觉得"麻烦"）。

## 六、一句话总结

让 AI 帮你写代码不难。让 AI 帮你写完之后，**代码、文档、提交、发布都不滑坡**才难。

难点不在 prompt 调得多巧，在仓库本身的**工程基础设施**够不够刚 —— hook、CI、sync 脚本、branch 策略、凭证体系，这些才是 agent 真正能"听话"的原因。

agent 不是宠物，不是同事，是工具。工具要有手柄、有刹车、有保险丝 —— harness 就是这些。
