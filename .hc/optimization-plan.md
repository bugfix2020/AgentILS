# Agent-Gate 项目优化方案（概要）

> 位置：.hc/optimization-plan.md
> 说明：列出建议优化项、优先级和预计耗时（粗估）。

## 快速结论（3 个高优先级切入点）
- **增加自动化测试与 CI** — 优先级：高；预计耗时：16-24 小时（1-3 天）。
- **移除阻塞式同步 IO、改为异步持久层** — 优先级：高；预计耗时：8-24 小时（1-2 天，依据复杂度）。
- **为工具与网关增加严格输入验证与统一错误处理** — 优先级：高；预计耗时：8-16 小时（1-2 天）。

## 详细优化条目

1. 增设自动化测试与 CI（高）
- 问题：仓库缺乏系统性单元/集成测试，仅存在脚本式测试。
- 建议：引入 `vitest` 或 `jest`、添加针对 `store`、`gateway`、`tools` 的单元测试；建立 GitHub Actions 流水线（lint、build、test、typecheck）。
- 依赖/风险：需要补充可注入的接口以便测试（例如 MemoryStore 的 mock）。
- 估时：16-24 小时。

2. 异步化并稳健化持久化层（json-persistence）（高）
- 问题：当前使用 `readFileSync` / `writeFileSync` 等同步 API，会阻塞事件循环；持久化缺乏事务或写入原子性。
- 建议：改为 `fs.promises` 或异步 API；支持批量/队列写入与写时原子替换（tmp file + rename）；可选：将 JSON 持久化替换为轻量级 DB（sqlite 或 LevelDB）以提高并发与可靠性。
- 估时：8-24 小时（快速异步化 4-8h；考虑切换到 sqlite 需要 1-2 天）。

3. 工具输入校验与统一错误/结果类型（高）
- 问题：部分工具返回自由文本 JSON，缺少统一的错误/成功格式与严格 schema 校验。
- 建议：对所有 tool 注册使用 Zod/JSON Schema 严格校验输入与输出；定义 `ToolResult` 统一结构；捕获并记录异常，返回可机器解析的错误码和提示。
- 估时：8-16 小时。

4. 网关（Gateway）层增强（中）
- 问题：入口逻辑较多，验证和审计散落，错误处理与可测性有提升空间。
- 建议：抽取验证/鉴权/配额检查为独立中间件函数，增加单元测试覆盖；集中化审计日志接口并规范事件 schema。
- 估时：8-12 小时。

5. 增加监控与度量（中）
- 问题：缺少运行时指标，难以定位性能瓶颈（例如工具调用次数、持久化延迟）。
- 建议：集成轻量指标（prom-client）并导出 /metrics，或在云环境接入 Datadog/Prometheus；记录关键执行业务指标与错误率。
- 估时：8-16 小时。

6. 内存存储与持久化的一致性保障（中）
- 问题：MemoryStore 与 JSON 持久化之间可能存在数据竞争或丢失风险。
- 建议：引入写入队列与持久化确认机制（ack），或在持久化时引入乐观锁/版本号以避免覆盖。添加恢复/迁移脚本。
- 估时：8-16 小时。

7. 代码质量与类型严格化（中）
- 问题：需确认是否开启 `tsconfig` 中最严格的检查与 ESLint 规则。
- 建议：启用 `strict: true`，补全类型定义，运行 `eslint` + `prettier`；修复 `any`/隐式类型。添加 `npm run typecheck`。
- 估时：8-12 小时。

8. 快速可执行的“快速胜利”项（低 -> 可在半天内完成）
- 添加 `npm` 脚本：`test`, `lint`, `typecheck`, `start:dev`（估时 1-2 小时）。
- 添加 ESLint + Prettier 配置并修复自动可修复问题（估时 2-4 小时）。
- 在 `scripts/` 下加入 CI 友好的集成测试脚本（估时 2-4 小时）。

9. 开发者体验与文档（低）
- 建议：补充 `CONTRIBUTING.md`、本地开发指南、运行与调试样例；更新 README 中的架构图与重点模块说明。
- 估时：4-8 小时。

10. 基础设施（低）
- 建议：提供 `Dockerfile` 与 `docker-compose`（可选），便于在 CI/容器中运行；或提供 `devcontainer`。
- 估时：4-8 小时。

## 实施优先级建议（短期路线图）
- 周期 0（0-2 天，快速落地）
  - 建立 CI（lint/build/typecheck）(4-8h)
  - 添加基础单元测试框架与示例测试（4-8h）
  - 将 json-persistence 异步化（4-8h 快速版）

- 周期 1（3-7 天，质量与可靠性）
  - 为关键模块（gateway、store、tools）补齐单元与集成测试（2-3 天）
  - 统一工具输入输出 schema 与错误处理（1-2 天）
  - 增加基本监控指标（1-2 天）

- 周期 2（7 天以上，扩展与硬化）
  - 引入 sqlite 或小型 DB 进行持久化替换并迁移数据（2-4 天）
  - 性能分析并针对热点优化（2-3 天）

## 风险与注意事项
- 部分改动（例如切换持久化层）会改变数据格式与迁移策略，需谨慎规划并提供回滚。
- 增加严格校验后可能暴露更多边界异常，需要协调调用方兼容处理。

## 建议的第一步（最小可行改进）
1. 在 repo 中加入 GitHub Actions：`ci.yml`（lint、build、typecheck、test）。
2. 将 `json-persistence` 改为 `fs.promises` 异步实现（并加入写入原子替换）。
3. 为 `MemoryStore` 添加一个或两个示例单元测试，验证核心行为。
